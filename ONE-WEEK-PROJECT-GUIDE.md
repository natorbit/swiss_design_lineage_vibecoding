## üéØ Vibe Coding Best Practices

### Working with AI in VS Code + Copilot

**Watch AI work:**
- Copilot suggests code as you (or AI chat) types
- Suggestions appear in gray text
- Press **Tab** to accept, **ESC** to reject

**When to stop AI:**
- Suggestions duplicate existing code
- Breaking accessibility (removing alt text, semantic HTML)
- Overcomplicating simple solutions
- Ignoring mobile considerations
- Resetting styles you already set
- Creating inconsistent patterns

**How to stop:**
- Press **ESC** to reject suggestion
- **CMD+Z** (Mac) or **CTRL+Z** (Windows) to undo
- Tell AI: "Stop. That's going in the wrong direction."

### Using AI as Different Experts

Don't just ask generic questions. Use AI as specialized experts:

**As UX Designer:**
> "Acting as a UX designer specializing in [your style], review this page for: information hierarchy, visual flow, interaction patterns, and accessibility. Compare to [style] best practices."

**As Accessibility Specialist:**
> "Acting as an accessibility specialist, audit this page for WCAG 2.1 AA compliance. Check: color contrast, keyboard navigation, semantic HTML, ARIA labels, screen reader compatibility."

**As Literary Critic:**
> "Acting as a literary critic, review this blog post for: clarity, engagement, structure, voice consistency, and readability. Suggest specific improvements."

**As Performance Engineer:**
> "Acting as a performance engineer, analyze this site's load time, asset sizes, and rendering efficiency. Identify specific optimizations."

**As [Style] Design Expert:**
> "Acting as a [your style] design expert, critique every aspect of this page's visual design. What's authentic? What's not? What would [famous practitioner] change?"

### The Screenshot Workflow

AI cannot see your site. Screenshots are essential.

**When to screenshot:**
- After every significant change
- Before every design review
- When something looks wrong
- For mobile responsiveness checks
- To document progress

**How to share with AI:**
- Take screenshot (full page if possible)
- Share in chat with specific question
- Include reference images for comparison
- Ask AI to describe what it sees (verify it understands)

**Good screenshot prompts:**
- "Here's the homepage hero section. Compare to this reference image. What's different?"
- "This mobile layout feels cramped. Analyze the spacing and suggest improvements."
- "The typography hierarchy isn't working. What specifically is wrong?"

### Iterative Design Reviews

**Never accept first draft.** Review everything 2-3 times minimum.

**First review:** Broad issues (layout, colors, typography scale)  
**Second review:** Specific sections (hero, navigation, components)  
**Third review:** Details (spacing, alignment, consistency)

**Each review cycle:**
1. Screenshot current state
2. Compare to references
3. Identify specific problems
4. Describe changes needed
5. AI implements
6. Screenshot again
7. Repeat until excellent

### Managing Context Windows

AI has limited memory. Control context by organizing conversations:

**Separate chats for different purposes:**
- **Chat 1:** Initial extraction and planning
- **Chat 2:** Sprint 1 execution
- **Chat 3:** Sprint 2 execution
- **Chat 4:** Sprint 3 execution
- **Chat 5:** Final polish and reflection

**In each new chat:**
- Upload key documents (`design-system.md`, `sprint-plan.md`)
- Reference previous decisions: "In Sprint 1, we decided to use [pattern]. Continue that approach."
- Keep focused on current task

**When context gets messy:**
- Start a new chat
- Reload essential documents
- Summarize where you are: "We've completed Sprints 1-2. Now starting Sprint 3 to add content and polish."

### Strategic Prompting

**Be specific:**
- ‚ùå "Make it look better"
- ‚úÖ "Increase line height to 1.6 and add 2rem of top padding"

**Provide context:**
- ‚ùå "Fix the navigation"
- ‚úÖ "The navigation feels too heavy for [style] aesthetics. [Style] typically uses lighter, more minimal navigation. Simplify this while keeping it functional."

**Reference sources:**
- ‚ùå "Use the design system"
- ‚úÖ "Reference `docs/design-system.md` section 3 (color palette) and apply those exact colors"

**Ask for reasoning:**
- "Why did you choose this approach?"
- "What's the [style] principle behind this decision?"
- "Explain the trade-offs here."

**Request alternatives:**
- "Show me 3 different ways to style this component"
- "What would happen if we used [alternative approach]?"

**Iterate strategically:**
- Start broad (overall layout)
- Then specific (individual components)
- Then details (spacing, alignment, polish)

### Knowing When Something's Wrong

**Trust your gut.** If something feels off, it probably is.

**Common signs:**
- "This looks too busy/cluttered"
- "The spacing feels wrong"
- "This doesn't match the references"
- "Mobile version is too cramped"
- "Colors are too aggressive/not enough contrast"
- "Typography hierarchy is confusing"

**When you sense a problem:**
1. Screenshot it
2. Compare to reference images
3. Describe what feels wrong (even if you can't articulate why precisely)
4. Ask AI to analyze: "What's causing this feeling of [crowdedness/imbalance/wrongness]?"
5. AI will identify specific technical issues
6. Fix and review again

**You don't need to know the technical solution.** You need to know something's wrong. AI finds the solution.
